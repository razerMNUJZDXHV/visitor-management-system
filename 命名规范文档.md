## 2. 命名规则

### 2.1 包名
- 采用小写字母，单词间用点分隔
- 遵循域名反转规则：`com.gpj.visitorsystem`
- 模块名使用小写字母：`controller`, `service`, `dto` 等
- 子模块名使用小写字母：`admin`, `wx`, `visitor`, `approver`, `security` 等

### 2.2 类名
- 使用大驼峰命名法（PascalCase）
- 控制器类：`[端类型][模块名]Controller.java`，如 `AdminUserController.java`、`WxVisitorAppointmentController.java`
- 服务类：`[端类型][模块名]Service.java`，如 `AdminUserService.java`、`WxApproverAppointmentService.java`
- 数据传输对象：`[端类型][业务含义]DTO.java`，如 `AdminLoginRequestDTO.java`、`SecurityAccessRecordDTO.java`
- 实体类：`[业务含义].java`，如 `User.java`、`Appointment.java`
- 工具类：`[功能]Util.java`，如 `JwtUtils.java`、`QRCodeUtil.java`
- 配置类：`[功能]Config.java`，如 `WebConfig.java`、`AppConfig.java`
- 拦截器：`[功能]Interceptor.java`，如 `AdminAuthInterceptor.java`、`WxAuthInterceptor.java`

### 2.3 方法名
- 使用小驼峰命名法（camelCase）
- 动词+名词形式，如 `createAppointment()`、`listPendingAppointments()`
- 保持方法名简洁明了，能准确表达方法的功能
- 遵循语义化命名，避免使用缩写

### 2.4 变量名
- 使用小驼峰命名法（camelCase）
- 变量名应具有描述性，能准确表达变量的含义
- 避免使用单个字母作为变量名（循环变量除外）
- 常量名使用全大写，单词间用下划线分隔，如 `MAX_PAGE_SIZE`

### 2.5 接口路径
- 使用小写字母，单词间用连字符分隔
- 遵循 RESTful 风格
- 路径前缀：
  - 管理后台：`/api/admin/...`
  - 微信小程序：`/api/wx/...`
  - 访客端：`/api/wx/visitor/...`
  - 审批人端：`/api/wx/approver/...`
  - 安保端：`/api/wx/security/...`

## 4. 注意事项

1. **一致性**：所有代码应遵循统一的命名规范，保持代码风格一致
2. **语义化**：命名应准确表达代码的功能和含义，避免使用模糊或误导性的命名
3. **简洁性**：命名应简洁明了，避免过长或过于复杂的名称
4. **规范性**：遵循 Java 语言的命名规范和最佳实践
5. **可读性**：良好的命名可以提高代码的可读性和可维护性

## 5. 总结

本命名规范旨在统一后端代码的命名风格，提高代码的可读性、可维护性和一致性。所有开发人员应严格遵循此规范进行代码开发和维护。
